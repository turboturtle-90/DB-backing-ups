# Домашнее задание к занятию «Резервное копирование» - `Смирнов Максим`

### Задание 1

Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования. Необходимо описать, какие варианты резервного копирования подходят в случаях:

1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.

1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных.

---

1.1. Для восстановления полного объема за предыдущий день нужен подход, гарантирующий RPO= 24часа. Этому требованию удовлетворяет, например, full backup, выполняемый ежедневно по ночам.

1.2. Для восстановления полного объема за час до поломки может применяться incrimental или differential backup strategy. При этом full backup будет также записываться один раз в сутки, а далее он будет комбинироваться с инкрементным/дифференциальным бэкапом.

1.3. Да, такой вариант возможен при наличии реплики, которая имеет повышенные требования по синхронизации данных с мастером. Однако, в соотвествии с определением репликации, это не является в полной мере резервной копией. Для данной ситуации реплика может работать в качестве резервной копии при утрате доступа к мастеру, но не при случайном / намеренном удалении данных, поскольку в таком случае данные с реплики исчезнут синхронно.

### Задание 2

Задание 2. PostgreSQL
2.1. С помощью официальной документации приведите пример команды резервирования данных и восстановления БД (pgdump/pgrestore).

2.2.* Возможно ли автоматизировать этот процесс? Если да, то как?

2.1. Обычно практикуется запись архивов, вызов команды с указанием -F c для резервного копирования

```
                                                         
pg_dump -h localhost -U my_user -F c -b -v -f /path-to-backup/my_db_backup.dump my_database

```
Расшифровка команды ниже:
-h localhost — адрес сервера БД
-U my_user — имя пользователя
-F c — Format: Custom. Создает сжатый бинарный архив
-b — включать в бэкап большие объекты (Large Objects)
-v — подробный вывод логов в консоль (Verbose)
-f ... — путь к файлу, куда сохранить бэкап
my_database — имя целевой базы данных.

Для восстановления из кастомного формата (-F c) используется утилита pg_restore:

```
                                                         
pg_restore -h localhost -U my_user -d my_database -v /path/to/backup/my_db_backup.dump


```

Расшифровка команды ниже:
-d my_database — имя базы данных, в которую мы восстанавливаем данные (она должна быть заранее создана).


2.2. Автоматизация процесса может выполняться путем написания bash скриптов, которые активируются с использованием chrone. 
Качественный процесс резервног копирования требует не только отписывания бэкапов, но и контроля этого процесса с целью убедиться, что бэкапы действительно сохранены, что места для новых достаточно, что файлы не повреждены и т.д.

В коммерческом применении этот путь требует большой кастомизации и высокой квалификации специалистов. Потому, очень часто как более простое решение испоьзуют коммерческие продукты которые автоматизируют процесс резервного копирования и контроля резервного копирования.
